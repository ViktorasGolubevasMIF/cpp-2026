    U6: Polimorfizmas

    Savaitės: 10-11

    Svoris: 1 balas (+ BONUS: 0,2)

    Terminas: 2026 gegužės 3

    👉 Užduočių Atlikimo ir Pateikimo Gidas

    🎯 Užduoties tikslas

    Išmokti naudoti virtualias funkcijas/metodus ir suprasti dinaminį susiejimą (dynamic binding); sukurti tikrą polimorfizmą — kai tas pats kvietimas s->area() veikia skirtingai priklausomai nuo tikrojo objekto tipo; suprasti abstrakčias klases ir virtualų destruktorių.

    📚 Mokymosi tikslai

    Atlikę šią užduotį, mokėsite:

    • ✅ Skirti statinį susiejimą (static binding) nuo dinaminio (dynamic binding)
    • ✅ Naudoti virtual raktažodį bazinėje klasėje
    • ✅ Naudoti override raktažodį paveldėtose klasėse
    • ✅ Sukurti naują paveldėtoją su override (perdengiančiais) metodais
    • ✅ Sukurti abstrakčią klasę su grynai virtualiais (pure virtual) metodais
    • ✅ Suprasti virtualaus destruktoriaus būtinybę

    📐 Užduoties formatas

    Žingsnių struktūra:

    • 6 žingsniai + BONUS
    • Žingsnis 1: Pilnas pavyzdys — problema (apšilimas)
    • Žingsniai 2–6: Vis daugiau // TODO: komentarų

    Kodo žymėjimai:

    • PAVYZDYS — pilnas veikiantis kodas (galite kopijuoti)
    • JŪSŲ KODAS — turite patys sukurti
    • // TODO: — komentarai nurodo, ką reikia padaryti

    Kontekstas: kur sustojome?

    U5 matėme problemą:

    Circle c(0, 0, 5.0);
Shape* s = &c;        // upcasting — veikia ✅

s->area();            // ← kodėl 0.0, jei c yra Circle?!

    area() grąžino 0.0 — kompiliatorius kvietė Shape::area(), ne Circle::area(). Tai statinis susiejimas (static binding): metodas parenkamas pagal rodyklės tipą (Shape*), ne pagal tikrąjį objekto tipą.

    U6 tikslas — tai ištaisyti vienu raktažodžiu: virtual.

    Logging'as

    Logging'as ([CTOR]/[DTOR]) išlieka visuose žingsniuose — stebėkite jį atidžiai. 6 žingsnyje jis taps įrodymu, kada destruktoriai kviečiami teisingai per delete Shape*?

    DALIS 1: virtual ir dynamic binding

    1 žingsnis: Static binding — problema

    Direktorija: U6/01/

    Motyvacija

    Nukopijuojame U5/05 kodą — be jokių pakeitimų. Tikslas: dar kartą pamatyti problemą ir suprasti kodėl ji egzistuoja, prieš ją sprendžiant.

    Kodas — PAVYZDYS

    Nukopijuokite iš U5/05/ visus failus (Point.h, Shape.h, Shape.cpp, Circle.h, Circle.cpp, Rectangle.h, Rectangle.cpp, Makefile).

    main.cpp: 

    #include <iostream>
#include <vector>
#include "Circle.h"
#include "Rectangle.h"

int main() {
    std::cout << "=== 1 žingsnis: Static binding problema ===\n\n";

    // --- A būdas: C stiliaus masyvas ---
    const int N = 2;
    Shape* arr[N];
    arr[0] = new Circle(0.0, 0.0, 3.0, "red");
    arr[1] = new Rectangle(1.0, 1.0, 4.0, 2.0, "blue");

    std::cout << "--- area() per Shape* arr[] ---\n";
    for (int i = 0; i < N; i++) {
        std::cout << "color=" << arr[i]->getColor()
                  << ", area=" << arr[i]->area() << "\n";
    }
    std::cout << "\n--- printInfo() per Shape* arr[] ---\n";
    for (int i = 0; i < N; i++) {
        arr[i]->printInfo();
    }
    for (int i = 0; i < N; i++) delete arr[i];

    // --- B būdas: vector<Shape*> ---
    std::vector<Shape*> shapes;
    shapes.push_back(new Circle(0.0, 0.0, 3.0, "red"));
    shapes.push_back(new Circle(1.0, 1.0, 5.0, "blue"));
    shapes.push_back(new Rectangle(2.0, 2.0, 4.0, 2.0, "green"));
    shapes.push_back(new Rectangle(3.0, 3.0, 6.0, 3.0, "black"));

    std::cout << "\n--- area() per vector<Shape*> ---\n";
    for (Shape* s : shapes) {
        std::cout << "color=" << s->getColor()
                  << ", area=" << s->area() << "\n";
    }
    std::cout << "\n--- printInfo() per vector<Shape*> ---\n";
    for (Shape* s : shapes) {
        s->printInfo();
    }
    for (Shape* s : shapes) delete s;

    return 0;
}

    Svarbu!

    a) Static binding — kompiliatorius parenka metodą kompiliavimo metu pagal rodyklės tipą (Shape*). Todėl visada kviečiama Shape::area(), net jei objektas yra Circle.

    b) Informacija neprarandama — tai ne slicing! Circle objektas heap'e yra pilnas, radius egzistuoja. Tik metodas parenkamas neteisingai.

    c) Abu būdai — ta pati problema — C masyvas ir vector elgiasi identiškai. virtual ištaisys abu.

    2 žingsnis: virtual — dynamic binding

    Direktorija: U6/02/

    Motyvacija

    Vienas raktažodis virtual pakeičia viską. Pridėjus jį Shape::area() — kompiliatorius nebesprendžia metodo kompiliavimo metu. Sprendimas atidedamas į vykdymo laiką: kviečiamas metodas pagal tikrąjį objekto tipą.

    Reikalavimai

    Nukopijuokite U6/01/ failus.

    Shape.h — JŪSŲ KODAS:

    Pridėkite virtual prie area() ir printInfo():

    class Shape {
protected:
    Point center;
    std::string color;

public:
    Shape(double x, double y, const std::string& color = "black");
    ~Shape();   // ← virtual destruktorius — apie tai 5 žingsnyje

    Point getCenter() const;
    std::string getColor() const;
    void setColor(const std::string& color);

    // TODO: pridėkite virtual prie area()
    double area() const;

    // TODO: pridėkite virtual prie printInfo()
    void printInfo() const;
};

    main.cpp — identiškas 1 žingsniui, tik pakeiskite preambulę:

    std::cout << "=== 2 žingsnis: virtual — dynamic binding ===\n\n";

    Svarbu!

    a) Vienas virtual Shape.h faile pakeitė viską — main.cpp nekeitėme. Tai polimorfizmo esmė: kodo, naudojančio objektus, keisti nereikia.

    b) sizeof padidėjo — virtual prideda vptr kiekviename objekte (plačiau - paskaitos medžiagoje). Patikrinkite: sizeof(Shape) dabar didesnis nei U5.

    c) Logging'as — stebėkite destruktorių eiliškumą. 6 žingsnyje jis taps dar svarbesnis.

    Eksperimentuokite — labai rekomenduojama!

    Išbandykite du dalinius variantus ir stebėkite kas keičiasi:

    Variantas A — tik area() virtual, printInfo() ne: 

    virtual double area() const;   // ← virtual
void printInfo() const;        // ← NE virtual
    Per Shape*: s->area() → teisingai ✅, bet s->printInfo() → Shape::printInfo() ❌. Tas pats objektas, ta pati rodyklė — vienas metodas veikia polimorfiškai, kitas ne.

    Variantas B — tik printInfo() virtual, area() ne: 

    double area() const;             // ← NE virtual
virtual void printInfo() const;  // ← virtual
    Per Shape*: s->printInfo() → Circle::printInfo() ✅ — spausdina teisingą plotą! Bet s->area() → Shape::area() → 0.0 ❌.

    Kodėl printInfo() rodo teisingą plotą net kai area() ne virtual? Nes Circle::printInfo() kviečia this->area() — o this yra Circle*, tad kviečiama Circle::area() tiesiogiai (statiškai susiejus), o ne "virtualiai" (dinamiškai susiejus)! Šis efektas/technika vadinamas visaip: „Virtualiu įėjimo tašku“ (per virtualią printInfo()), statiniu area() susiejimu dinaminiame printInfo() kontekste...

    3 žingsnis: override — saugiklis

    Direktorija: U6/03/

    Motyvacija

    Circle::area() jau perdengia (overrides) Shape::area() — net be jokio papildomo raktažodžio. Tai veikia. Bet C++11 įvedė override — ne dėl funkcionalumo, o dėl saugumo:

    // Be override — tyli klaida:
class Circle : public Shape {
    double area() const;   // ← ar tikrai perdengia? o jei parašėm area(int)?
};

// Su override — kompiliatorius patikrina:
class Circle : public Shape {
    double area() const override;  // ← garantija: tikrai perdengia Shape::area()
};

    Reikalavimai

    Nukopijuokite U6/02/ failus.

    Circle.h — JŪSŲ KODAS:

    class Circle : public Shape {
private:
    double radius;
public:
    Circle(double x, double y, double r,
           const std::string& color = "black");
    ~Circle();

    double getRadius() const;

    // TODO: pridėkite override prie area()
    double area() const;

    // TODO: pridėkite override prie printInfo()
    void printInfo() const;
};

    Rectangle.h — JŪSŲ KODAS (analogiškai):

    // TODO: pridėkite override prie area() ir printInfo()

    main.cpp — JŪSŲ KODAS:

    Pirmiausia — tyčinė klaida be override:

    // TODO: išbandykite tylią klaidą — pamiršę const:
class Shape {
    virtual double area() const;   // ← signatūra su const
};
class Circle : public Shape {
    double area();                 // ← pamiršta const — tai OVERLOAD, ne override!
    // TODO: pridėkite override ir pažiūrėkite į kompiliavimo klaidą:
    // double area() override;    // ← [NC]: "does not override any virtual function"
};

    Tada — pilnas main.cpp su vector<Shape*> ir C masyvu (kaip 2 žingsnyje).

    Svarbu!

    a) override — ne funkcionalumas, o saugiklis — be jo programa veikia, bet "tyliai" neiškviečiamas teisingas metodas. Su override — kompiliatorius garantuoja.

    b) Pamiršta const → OVERLOAD, ne override — skirtinga signatūra sukuria naują metodą. Tiesa gcc praneša, jei paprašyti: -Woverloaded-virtual.

    override vs overload — priminimas

    Detaliau — P08 paskaitoje. Esmė: override reikalauja identiškos signatūros. const yra signatūros dalis — pamiršus jį gauname naują perkrautą metodą, ne metodo perdengimą.

    6 žingsnis: Virtualus destruktorius

    Direktorija: U6/06/

    Motyvacija

    Iki šiol Shape destruktorius nebuvo virtual. Kas atsitinka kai kviečiame delete per Shape*? Logging'as parodys atsakymą — ir jis bus netikėtas.

    Reikalavimai

    Pirma — išbandykite be virtual destruktoriaus (kaip dabar). Paleiskite programą ir stebėkite logging'ą.

    Svarbu!

    a) Virtual destruktoriaus taisyklė: jei klasė turi bent vieną virtual metodą — destruktorius visada turi būti virtual. Be jo delete per Shape* → undefined behavior.

    b) Logging'as tai įrodo — skirtumas matomas akimirksniu. Be virtual: tik [Shape DTOR]. Su virtual: [Circle DTOR] → [Shape DTOR].

    BONUS: IDrawable interfeisas

    Direktorija: U6/BONUS/

    Motyvacija

    Shape abstrakti klasė turi ir duomenis (center, color) ir pure virtual metodus. Kartais norima grynesnio interfeiso — tik elgsena, jokių duomenų. Tai interfeiso (interface) idėja, kurią C++ realizuoja per abstrakčias klases be duomenų narių.

    Reikalavimai

    Nukopijuokite U6/05/ failus.

    IDrawable.h — JŪSŲ KODAS: 

    #pragma once

class IDrawable {
public:
    // TODO: grynai virtualus draw() metodas

    // TODO: virtualus destruktorius (kodėl būtinas?)
};

    Atnaujinkite Shape kad paveldėtų IDrawable: 

    // Shape.h:
class Shape : public IDrawable {
    // ...
    // TODO: implementuokite draw() — paprastumui, galite kviesti printInfo() ar kažkiek jį "pagražinti"
};

    main.cpp — JŪSŲ KODAS: 

    // TODO: pademonstruokite IDrawable per rodyklę:
IDrawable* d = new Circle(0, 0, 5.0);
d->draw();    // ← veikia per IDrawable*
delete d;

// TODO: vector<IDrawable*> su skirtingais objektais

    Svarbu!

    a) Interfeisas C++ — abstrakti klasė be duomenų narių, tik pure virtual metodai. Konvencija: pavadinimas pradedamas I (pvz. IDrawable, ISerializable).

    b) Kitose kalbose — Java ir C# turi atskirtą interface raktažodį. C++ tai realizuojama per abstrakčias klases.

    c) = default — virtual ~IDrawable() = default liepia kompiliatoriui sugeneruoti destruktorių. Tai modernesnė alternatyva tuščiam {}.

    💡 Patarimai

    1. virtual pakanka tik bazinėje klasėje — Circle::area() automatiškai tampa virtual net be raktažodžio. Bet override rekomenduojama visada rašyti aiškumo dėlei.

    2. Pure virtual ir konstruktorius — abstrakti klasė vis tiek gali turėti konstruktorių. Jis kviečiamas per paveldėtojų konstruktorių grandinę.

    3. #include <cmath> — būtina Triangle::area() dėl sqrt().

    4. Herono formulės patikrinimas — jei s*(s-a)*(s-b)*(s-c) < 0, kraštinės neteisingos. Galima mesti išimtį arba tiesiog ignoruoti (neprivaloma).
